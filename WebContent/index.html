MPI Critical Path Profiler


Overview

MPI is designed in such a way that each MPI function is defined as a weak symbol. That means that all MPI functions have an implementation within the MPI library. However, if the programmer overrides that implementation by creating their own version of the MPI function, that version will instead be Invoked.


Requirement

By Overriding the MPI Implementation of the functions during execution of the MPI Program, we will generate a MPI Task Graph which is a directed, weighted graph. There is an edge between every pair of consecutive vertices V and W. The edge weight is the wall clock time that has elapsed from the end of the MPI operation corresponding to V and the beginning of the MPI operation corresponding to W.


Results

After program execution (at MPI Finalize), the MPI critical path graph should then be displayed on the screen (Using GraphViz). Critical path is the longest path in execution. Using which the user can identify bottlenecks in execution flow.

  • C
  • MPI

Overview

The Router uses VNS which enables it to process and send real Ethernet frames, sending them over the network like a real router. The Router must handle the various functionalities that an actual router would. In case of a link failure the router uses PWOSPF to route traffic through the network.


Requirement

The router should be able to handle

  1. Perform sanity checks against all incoming packets
  2. Handle ARP messages (Request and Reply)
  3. Maintain an ARP Cache which refreshes every 30 sec.
  4. Handle ICMP packets to detect if a host is reachable or not
  5. Handle IP packets.
  6. Handle Ping packets to other systems in the topology or to its own(Router's) interfaces
  7. Handle Traceroute to display the path a packet takes to reach its destination.
  8. Handle link failures using PWOSPF to reroute network traffic. PWOSPF must be able to rebuild the correct routing table when a link goes down using Link State Updates
  9. Router must be able to download/play/view a music file, image and a video

Failure tolerance in Distributed Computing (rMPI)


Overview

Process replication, generally referred to as state machine replication, is a well-known technique for tolerating faults in systems that target high-availability. In this approach, a process's state is replicated such that if the process fails, its replica is available or can be generated to assume the original process's role without disturbing the other application processes.


Requirement

The implementation of rMPI must be able to handle both Mirrored and Parallel Protocol as specified in the paper.
The user will execute an MPI program on "N" Nodes, desiring an "N" Nodes replica ( Each primary node has a replica node ).
We will support only MPI Send, MPI Recv and MPI Barrier (dissolved into MPI Send and Recv).
We will also be supporting ANY_SOURCE.
A node will be "killed" (by a test programs) explicitly from a user program which makes a calls to MPI_Pcontrol. The single parameter will indicate the node to be killed.
Note: If a node is to be killed, the id will be in the range N to 2N-1. When a node fails the surviving partner node must take over the role of the failing partner for the rest of the application.


Results

Implemented the redundant Fault tolerance scheme for MPI programs using both Parallel and Mirred Protocol
Mirrored protocol, as per the specification , let A and B represent distinct MPI Nodes and A' and B' their replicas respectively.
In this protocol, each sender transmits duplicate messages to each of the destinations.
Similarly, receivers must post multiple receives for the duplicate messages, but only require one of those messages to arrive in order to progress.
This eases recovery after a failure, but doubles network bandwidth requirements.

Parallel protocol, In this approach, each replica has a single corresponding replica for each other rank (Node) with which it communicates in non-failure scenarios.
In the case of failure, one of the remaining replicas of a rank takes over sending and receiving for the failed node.
This failure detection requires frequent message-based interaction with the reliability system on current systems.
As a result, the parallel protocol will initiate approximately double the number of messages for each send operation.
These extra messages contain MPI envelope information and are small. Hence the parallel protocol reduces network bandwidth requirements for an increased number of short messages.
